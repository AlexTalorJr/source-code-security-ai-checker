   - Re-evaluate PHP Security Checker: dependency scanning for PHP, works well

   - Recommendation: NOT recommended. Semgrep already covers JS/TS more thoroughly. 14 rules don't justify Node.js dependency. Note this clearly.
   - Comparison matrix: Semgrep vs eslint-plugin-security (Semgrep wins on coverage, no extra runtime)

4. **Go** — Current: Semgrep only. Recommended addition: gosec.
   - gosec research card: CLI `gosec -fmt=json ./...`, Docker single Go binary ~15MB, SARIF yes (`-fmt=sarif`), effort S, 70+ security checks (SQL injection, file traversal, hardcoded creds, crypto misuse)

   - cargo-audit research card: CLI `cargo-audit audit --json`, Docker `cargo install cargo-audit` ~10MB, NO SARIF (JSON only), effort S, RustSec advisory database
   - Note: SCA only (dependency vulnerabilities), not SAST. Semgrep handles Rust SAST.
   - Requires Cargo.lock file present
   - Config snippet: `scanners: cargo_audit: enabled: "auto", timeout: 60, extra_args: []`
   - FindingSchema mapping: rule_id from advisory `id` (e.g. RUSTSEC-2021-0001), severity from advisory `cvss` score mapped to LOW/MEDIUM/HIGH/CRITICAL, file_path as "Cargo.lock", snippet from affected crate + version
   - Comparison matrix: Semgrep (SAST) + cargo-audit (SCA) — complementary, not overlapping

6. **Java** — Current: Semgrep only. Evaluated: SpotBugs + FindSecBugs.

   - CRITICAL concern: requires compiled .class files, not source code. Scanner currently works on source directories only. Would need a build step before scanning. This is a major integration challenge.
   - Config snippet: `scanners: spotbugs: enabled: "auto", timeout: 300, extra_args: ["-effort:max", "-threshold:medium"]`
   - False positive reduction: use `-threshold:medium` to skip low-confidence, `-effort:max` for best detection
   - Comparison matrix: Semgrep vs SpotBugs+FindSecBugs (Semgrep: source-level, no compilation needed, lighter. SpotBugs: deeper bytecode analysis, 144 security types, but heavy + compilation required)

   - Requires .sln/.csproj files, Roslyn-based analysis
   - Config snippet: `scanners: security_code_scan: enabled: "auto", timeout: 180, extra_args: []`

8. **C/C++** — Current: Cppcheck. Recommendation: Keep existing.

- Grype research card: CLI `grype dir:/path -o json`, Docker single binary ~30MB, SARIF yes (`-o sarif`), effort S, CVSS + EPSS + KEV risk scoring. Better risk prioritization than Trivy but narrower scope.
  - Config snippet: `scanners: grype: enabled: "auto", timeout: 120, extra_args: []`
  - FindingSchema mapping: rule_id from vulnerability `id` (CVE-xxxx-xxxx), severity from `severity` field (Critical/High/Medium/Low/Negligible), file_path from artifact `name`, snippet from `matchDetails`

- Comparison matrix with columns: Tool | Scope | Risk Scoring | Output Formats | SARIF | Docker Size | Effort | Recommendation
  - Trivy: Keep as primary (broadest scope)
  - Grype: Consider as complement for risk prioritization (EPSS + KEV scoring)
  - OWASP Dep-Check: Skip (Trivy covers this, XML output, higher false positives)

**2. DAST Feasibility Assessment section:**
- Header: `## DAST (Dynamic Application Security Testing) Feasibility`
- Opening note: DAST requires a running application, fundamentally different from SAST/SCA. Cannot use the same ScannerAdapter pattern without modification. May need a separate DastAdapter base class.
- ZAP research card: CLI `zap-cli quick-scan --self-contained https://target`, Docker requires JVM + full ZAP ~500MB+, SARIF via community plugins, effort L. Most comprehensive but hardest to integrate.
- Nuclei research card: CLI `nuclei -u https://target -jsonl`, Docker single binary ~30MB, SARIF yes, effort M, 11,000+ community templates. Template-based, CLI-friendly, fast.
  - Config snippet: `scanners: nuclei: enabled: false, timeout: 300, extra_args: ["-severity", "medium,high,critical"]`
- Nikto: NOT recommended. Server-level only, limited value compared to Nuclei.
- Integration feasibility summary: Nuclei is the best DAST candidate (CLI-friendly, lightweight, fast). ZAP is most thorough but massive footprint. Both require a running application target URL, which the current scanner doesn't handle.
- Comparison matrix: Tool | Type | Best For | Docker Size | SARIF | Effort | Recommendation
  - Nuclei: recommended if DAST pursued (M effort, template-based)
  - ZAP: defer (L effort, massive footprint)
  - Nikto: skip (limited value)

**3. Secrets Scanning Comparison section:**
- Header: `## Secrets Scanning Comparison`
- Re-evaluate Gitleaks (current): version 8.30.0, fast regex-only scanning, ~15MB binary, no network calls, higher false positives due to no verification
- TruffleHog comparison: ~50MB binary, pattern matching + active credential verification, slower, lower false positives, broader source support (S3, Docker, Slack)
- Verdict: Keep Gitleaks as primary. Speed and simplicity win for CI/CD integration. TruffleHog's verification is valuable but adds complexity and network dependency. Consider TruffleHog as optional complement for deep audits.
- Comparison matrix: Feature | Gitleaks | TruffleHog (columns: Speed, Detection method, Scope, False positives, Docker size, Recommendation)
