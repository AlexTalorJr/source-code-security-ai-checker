# Phase 5 — UI Design Contract

> Visual and interaction contract for the server-rendered dashboard, login page, notification templates (Slack Block Kit, HTML email), and API error responses. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|

| Preset | not applicable |

## Spacing Scale

Declared values (must be multiples of 4). Identical to existing report.html.j2 CSS custom properties:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, badge padding |
| sm | 8px | Compact element spacing, form field gaps, table cell padding |
| md | 16px | Default element spacing, card padding, section gaps |
| lg | 24px | Section padding, page margins, form group spacing |
| xl | 32px | Layout gaps between major sections |
| 2xl | 48px | Page-level section breaks |

Exceptions: 44px minimum touch target on dashboard action buttons (suppress/unsuppress) for usability.

|-----------|-------------|----------|

## Interaction Contracts

### Dashboard Login Flow

1. User navigates to `/dashboard/*` without valid session cookie
2. Redirect 302 to `/dashboard/login`
3. User enters API key, clicks "Log In"
4. Server validates key via `secrets.compare_digest` against `settings.api_key`
5. On success: set `scanner_session` cookie (SHA-256 hash of API key), redirect to `/dashboard/history`
6. On failure: re-render login page with error message "Invalid API key. Check your SCANNER_API_KEY configuration."

### Scan Trigger Flow (Dashboard)

1. User clicks "Start New Scan" on history page
2. Form appears (inline or modal) with path/repo_url/branch fields
3. User fills fields, clicks "Start Scan"
4. Dashboard POSTs to `/api/scans` with session-derived API key
5. On 202: redirect to `/dashboard/scans/{id}` (detail page)
6. Detail page shows "Queued" or "Running..." status with `<meta http-equiv="refresh" content="5">`
7. When status becomes "completed", page shows full results (no more auto-refresh)

### False Positive Suppression Flow

1. User views finding in detail page Findings tab
2. Clicks "Suppress" button on finding row
3. Browser confirmation dialog: "Mark this finding as false positive? It will be excluded from all future scans and quality gate decisions."
4. On confirm: PUT `/api/findings/{fingerprint}/suppress` with optional reason
5. Finding moves from Findings tab to Suppressed tab
6. To reverse: click "Restore" on Suppressed tab, confirm dialog, DELETE request

## Email Template Layout

Structure (inline CSS for email client compatibility):

1. **Header bar**: 48px height, accent color (`#0d6efd`) background, white text "aipix-security-scanner"
2. **Gate banner**: Full width, green (`#d4edda`) or red (`#f8d7da`) background, 28px/600 gate status text
3. **Metadata row**: Branch, commit, duration in 12px text-secondary
4. **Severity table**: 4-column table (Severity, Count, color indicator), 14px body, alternating row backgrounds
5. **Delta section** (if available): +N new (red), -M fixed (green) badges
6. **CTA button**: Accent-colored button linking to dashboard scan detail page
7. **Footer**: 12px text-secondary, "Sent by aipix-security-scanner. Configure notifications in config.yml."

All styles must be inline (no `<style>` block) for email client compatibility.
