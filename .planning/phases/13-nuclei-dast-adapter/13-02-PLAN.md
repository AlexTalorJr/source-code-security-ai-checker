    Add Nuclei v3.7.1 binary installation to the Dockerfile AFTER the cargo-audit install block (around line 70) and BEFORE the non-root user creation (line 72 `RUN groupadd`).

    Insert this block:

    ```dockerfile
    # Install nuclei (DAST scanner) -- uses zip format, not tar.gz
    ARG TARGETARCH
    RUN curl -sSL "https://github.com/projectdiscovery/nuclei/releases/download/v3.7.1/nuclei_3.7.1_linux_${TARGETARCH}.zip" \
        -o /tmp/nuclei.zip && \
        unzip -o /tmp/nuclei.zip -d /usr/local/bin nuclei && \
        chmod +x /usr/local/bin/nuclei && \
        rm /tmp/nuclei.zip

    # Bake Nuclei templates into image (runs as root, before USER switch)
    # Templates stored in /root/.local/nuclei-templates/ then copied for scanner user
    RUN nuclei -update-templates && \
        mkdir -p /home/scanner/.local && \
        cp -r /root/.local/nuclei-templates /home/scanner/.local/nuclei-templates
    ```

    After the `RUN groupadd ... && useradd ...` line, add ownership of nuclei templates:
    ```dockerfile
    RUN chown -R scanner:scanner /home/scanner/.local
    ```

    **Critical notes per RESEARCH.md pitfalls:**
    - Use `unzip` NOT `tar xz` -- Nuclei releases are zip files (Pitfall 7)
    - `unzip` is already available from the PHP tools apt-get block
    - Use `TARGETARCH` directly (Docker BuildKit provides `amd64` or `arm64`) -- Nuclei release filenames use these exact strings
    - Run `nuclei -update-templates` BEFORE `USER scanner` switch (Pitfall 6) -- templates download to root's home
    - Copy templates to scanner user's home so they are accessible at runtime (Pitfall 5 and 6)
    - The `ARG TARGETARCH` line is needed to make the build arg available in subsequent RUN commands

    ```yaml

        timeout: 300
        extra_args: []
        adapter_class: "scanner.adapters.nuclei.NucleiAdapter"

    ```

    Key config decisions per CONTEXT.md:
    - `enabled: true` (NOT "auto") because DAST is not language-dependent. It only runs when target_url is provided via orchestrator routing, so "auto" language detection is irrelevant.
    - `timeout: 300` because DAST scans over HTTP are slower than local SAST analysis
    - `extra_args: []` -- users can add `-tags cve,exposure` or `-severity medium,high,critical` as needed
    - `adapter_class: "scanner.adapters.nuclei.NucleiAdapter"` -- the dotted path the registry uses to import
    - `languages: []` -- not language-dependent

    Also check if existing scanner entries already have `adapter_class`. If they do NOT have `adapter_class` fields in config.yml.example, that is fine -- the registry loads them via the config.yml the user actually uses, and the example file may be outdated. Only add `adapter_class` to the nuclei entry.
