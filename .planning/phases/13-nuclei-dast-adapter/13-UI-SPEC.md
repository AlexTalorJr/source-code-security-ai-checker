# Phase 13 — UI Design Contract

> Visual and interaction contract for the Nuclei DAST Adapter phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Assessment

**This phase is backend-only.** No new dashboard pages, components, or templates are introduced. Nuclei DAST findings flow through the existing FindingSchema pipeline and render automatically in existing HTML/PDF report templates via the dynamic `all_tools` loop and severity badge system.

**Zero frontend files are created or modified in this phase.**

The contract below documents the existing design tokens that Nuclei findings will render through, ensuring downstream consumers (planner, executor, auditor) understand the visual context without ambiguity.

## Copywriting Contract

No new UI copy is introduced in this phase. All user-facing text flows through the API layer (JSON responses) and existing report templates.

| Element | Copy | Source |
|---------|------|--------|
| API validation error (target_url + path conflict) | "target_url cannot be combined with 'path' or 'repo_url'." | RESEARCH.md Pattern 3 |
| API validation error (no target provided) | "Provide 'path', 'repo_url', or 'target_url'." | RESEARCH.md Pattern 3 |
| Scanner execution error | Propagated via existing ScannerExecutionError pattern | Existing adapter contract |
| Report tool badge | "nuclei" (lowercase, rendered in existing tool filter) | Existing all_tools loop |
| Report finding title | Nuclei template name (e.g. "nginx version detect") | Mapped from info.name field |
| Report finding file_path | Target URL (e.g. "http://example.com/") | Mapped from matched-at field |

No primary CTA, empty state, or destructive confirmation copy needed -- this phase adds no interactive UI surfaces.

## Existing Rendering Contract (How Nuclei Findings Appear)

Nuclei findings render through these existing report mechanisms with zero template changes:

1. **Tool filter sidebar**: `report.html.j2` iterates `all_tools` and creates a checkbox filter per tool. "nuclei" appears automatically.
2. **Finding cards**: Each finding renders with tool badge, severity badge, title, file_path (URL for DAST), and snippet. All fields populated by NucleiAdapter's FindingSchema mapping.
3. **Severity grouping**: Findings grouped by severity in the report -- Nuclei findings intersperse with SAST findings under the same severity headings.
4. **Quality gate banner**: Gate status computed from all findings including Nuclei -- no gate logic changes needed.
5. **PDF report**: Uses the same template structure as HTML -- Nuclei findings included automatically.
