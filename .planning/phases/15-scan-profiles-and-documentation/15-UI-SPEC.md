## Spacing Scale

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, checkbox-to-label gap, badge margin |
| sm | 8px | Compact element spacing, button padding vertical, input padding, table cell padding, navbar gap |
| md | 16px | Default element spacing, card padding, form-group margin-bottom, form-row gap, button padding horizontal |
| lg | 24px | Section padding, content area padding, navbar horizontal padding, scanner-grid gap, page-heading margin-bottom |

| 2xl | 48px | Empty state vertical padding |
| 3xl | 64px | Not currently used -- available for page-level spacing |

**Source:** Extracted from base.html.j2 and scanners.html.j2 existing CSS.

|------|------|--------|-------------|
| Body | 14px | 400 (regular) | 1.5 |
| Label | 12px | 600 (semibold) | 1.4 |
| Heading | 20px | 600 (semibold) | 1.2 |

Font sizes used: 10px (role badge only), 11px (scanner name badge in profile cards), 12px (label, badge, table header), 14px (body, input, button, nav link), 20px (page heading, empty state heading).

## Component Inventory (Phase 15 Additions)

### 1. Profiles Tab Button

- Third button in existing `.tab-bar` alongside "Scanners" and "YAML Editor"
- Label: **Profiles**
- Same styling as existing tab buttons: `.tab-btn`, active state `.tab-btn--active`
- Active state: `--color-text` text + `--color-accent` bottom border (2px)
- Inactive state: `--color-text-secondary` text + transparent bottom border

### 2. Profile Card Grid

- Container: reuse `.scanner-grid` class (auto-fill, minmax(320px, 1fr), 24px gap)
- Card: reuse `.scanner-card` class (1px border, 8px radius, 16px padding, hover shadow)
- Card title row: reuse `.card-title-row` (flex, space-between)
- Card title: profile name, `.card-title` (14px, weight 600)
- Description line: reuse `.card-languages` class (12px, --color-text-secondary, margin-top 4px)
- Scanner badges: `<span class="badge badge--success">` per scanner name, font-size 11px, margin 2px
- Edit/Delete buttons: bottom-right of card, `.btn--small` for edit, `.btn--small.btn--destructive` for delete

### 3. Profile Edit Form (Inline Expanded)

- Uses `.card-settings` / `.card-settings--open` pattern (same as scanner card expand)
- Appears below the card content, inside the card element
- Background: `--color-surface`
- Border-top: 1px solid `--color-border`
- Padding: 16px
- Negative margin: 16px -16px -16px -16px (to fill card width)
- Fields:
  - **Name** input: `.form-input`, disabled on edit (name is the key), enabled on create
  - **Description** input: `.form-input`, type text, placeholder "Brief description of this profile"
  - **Scanner checklist**: one row per scanner, each row contains:
    - Checkbox input (native HTML checkbox, width auto)
    - Scanner name label (14px, regular weight)
    - Optional timeout override: number input (inline, width 80px), min 30, max 900, placeholder from base config
  - Checklist container: max-height 400px, overflow-y auto (scrollable if many scanners)
- Action buttons: `.btn-row` (flex, justify-content flex-end, margin-top 12px)
  - "Save Profile" button: `.btn .btn--accent`
  - "Cancel" button: `.btn` with default styling (no accent, just border)

### 4. New Profile Button

- Positioned above the profile grid
- Label: **New Profile**
- Class: `.btn .btn--accent`
- Behavior: expands a blank profile card at the top of the grid with the inline form open

### 5. Profile Dropdown on Scan Form

- Added to `.start-scan-form` on history.html.j2
- Position: new `.form-row` between the existing checkbox row and the "Start Scan" button
- Element: `<select>` with `.form-select` class (already defined in base.html.j2)
- Label: "Scan Profile" (12px, weight 600, --color-text-secondary)
- Options: "(No profile)" as default empty value, then each profile name from config
- Width: flex 1, min-width 200px (matches other form fields)

### 6. Profile Name in Scan History Table

- New `<th>` column: "Profile" between "Branch" and "Status" columns
- Cell content: profile name string, or "-" if no profile was used
- Same cell styling as other table cells (14px, 8px padding)

### 7. Delete Profile Confirmation

- Native browser `confirm()` dialog (consistent with the project's no-framework approach)
- No custom modal component needed

## Interaction Contracts

### Tab Switching (Profiles tab)

- Click "Profiles" tab button: hide Cards panel, hide YAML panel, show Profiles panel
- Click "Scanners" tab button: `location.reload()` (same as existing behavior when switching away from YAML)
- Click "YAML Editor" tab button: hide other panels, fetch fresh YAML, show editor (existing behavior)
- Only one tab panel visible at a time

### Profile Card Expand/Collapse

- Click card body (not buttons): toggle `.card-settings--open` on that card's settings panel
- Accordion behavior: expanding one card collapses all others (same as scanner cards in Phase 14)
- Click Save/Cancel/Delete buttons: `event.stopPropagation()` to prevent card toggle

### New Profile Flow

1. User clicks "New Profile" button
2. A blank card appears at the top of the grid with form already expanded
3. Name field is editable (empty, focused)
4. All scanner checkboxes unchecked by default
5. User fills name, optionally description, checks desired scanners, optionally sets timeout overrides
6. Click "Save Profile": POST to `/api/config/profiles`, on success show success alert and collapse form, reload profile list
7. Click "Cancel": remove the blank card from DOM

### Edit Profile Flow

1. User clicks on an existing profile card to expand it
2. Form shows current values: name (disabled), description, scanner checkboxes pre-checked, timeout overrides pre-filled
3. User modifies values
4. Click "Save Profile": PUT to `/api/config/profiles/{name}`, on success show success alert and collapse
5. Click "Cancel": collapse form, discard changes

### Delete Profile Flow

1. User clicks "Delete" button on a profile card
2. Native `confirm()` dialog with the destructive confirmation copy
3. If confirmed: DELETE to `/api/config/profiles/{name}`, on success show success alert and remove card from DOM
4. If cancelled: no action

### Alert Behavior

- Success alerts: auto-dismiss after 3000ms (matches Phase 14 pattern)
- Error alerts: persist until next action or page reload
- Alert placement: inside the card settings panel for card-level alerts, above the grid for page-level alerts
