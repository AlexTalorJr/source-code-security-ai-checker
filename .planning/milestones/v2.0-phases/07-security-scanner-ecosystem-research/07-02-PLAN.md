**1. SARIF Evaluation section:**
- Header: `## SARIF Evaluation`
- Table of SARIF support across all 13 tools researched:
  - SARIF supported (8): Semgrep/Opengrep, gosec, Bandit, Brakeman, SpotBugs, security-code-scan, Trivy, Grype
  - SARIF NOT supported (5): cargo-audit (JSON only), Cppcheck (XML only), eslint-plugin-security (ESLint JSON), Gitleaks (custom JSON), Checkov (custom JSON)
- Recommendation: Introduce an optional SARIF parsing path alongside existing JSON parsing. Create a shared `parse_sarif()` helper function that maps SARIF `results[].ruleId`, `results[].level`, `results[].locations[].physicalLocation` to FindingSchema fields. SARIF-capable tools can use this helper to reduce per-adapter boilerplate. Non-SARIF tools keep custom parsers.
- Effort estimate: M — shared helper is straightforward (SARIF is just JSON with known schema), but retrofitting existing adapters is optional
- Note: SARIF v2.1.0 is the OASIS standard. Schema is well-documented. The `sarif-tools` pip package exists but direct JSON parsing is sufficient.

**2. Plugin Architecture Patterns section:**
- Header: `## Plugin Architecture Patterns`
- Comparison table of 4 approaches:
  - stevedore: setuptools entry_points, battle-tested, namespace isolation, lazy loading. Overkill for <20 plugins. Extra dependency.
  - importlib entry_points: Python stdlib 3.9+, no extra deps, standard pattern. Less helper infrastructure. Good fit.
  - Config-driven registry: YAML config lists adapter class paths. Simplest, matches existing config.yml pattern. No dynamic discovery. BEST FIT.
  - Directory scanning: Auto-discover adapters in folder. Zero config for new adapters. Implicit, harder to control ordering. Acceptable.
- Recommendation: **Config-driven registry** pattern. Justification:
  - Project already uses config.yml for scanner enablement
  - Extend ScannersConfig to accept `adapter_class` field per scanner entry mapping tool_name to Python class path (e.g., `scanner.adapters.gosec.GosecAdapter`)
  - New scanner workflow: (1) create Python file in `src/scanner/adapters/`, (2) add entry in config.yml with `adapter_class`, `enabled`, `timeout`, `extra_args`, (3) add language mapping in `SCANNER_LANGUAGES`
  - No stevedore dependency needed — project has <20 scanners
  - Example extended config:
    ```yaml
    scanners:
      gosec:
        adapter_class: "scanner.adapters.gosec.GosecAdapter"
        enabled: "auto"
        timeout: 120
        extra_args: ["-exclude=G101"]
    ```
  - Migration path: Current hard-coded ALL_ADAPTERS list can coexist with config-driven registration during transition

**3. Orchestration Improvements section:**
- Header: `## Orchestration Improvements`
- Sub-section "Parallel Execution": Already implemented via `asyncio.gather` in orchestrator.py. Current pattern is good. No changes needed.
- Sub-section "Incremental Scanning (Changed-Files-Only)":
  - Concept: Pass a file list to scanners instead of entire directory for faster CI/CD feedback
  - Tools that support file-list input:
    - Semgrep: `--include` flag with file patterns
    - Bandit: `-f` with file list, or pass individual files as args
    - gosec: accepts path arguments for specific packages
    - Brakeman: `--only-files` flag
    - Gitleaks: `--no-git` with specific paths
  - Tools that require full project context (CANNOT do incremental safely):
    - SpotBugs: needs compiled bytecode of full project
    - Trivy: needs complete lockfiles for dependency resolution
    - Checkov: needs full IaC context for relationship analysis
    - cargo-audit: needs Cargo.lock (project-level, not file-level)
  - Recommendation: Implement opt-in incremental mode. Orchestrator accepts optional `changed_files: list[str]` parameter. For tools that support file-list input, pass only changed files. For tools that need full context, fall back to full scan. Effort: M.
- Sub-section "Deduplication Improvements":
  - Current: fingerprint-based deduplication (file_path + rule_id + snippet)
  - Challenge: Multiple tools reporting same vulnerability with different rule_ids (e.g., Semgrep SQL injection rule vs Bandit SQL injection rule)
  - Recommendation: Add a cross-tool deduplication layer that groups findings by (file_path, line_range, vulnerability_class). Effort: M. Defer to implementation phase.
